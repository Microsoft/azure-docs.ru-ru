# <a name="secure-remote-access-to-virtual-machines-in-azure-active-directory-domain-services"></a>Безопасный удаленный доступ к виртуальным машинам в доменных службах Azure Active Directory

Чтобы защитить удаленный доступ к виртуальным машинам, которые работают в управляемом домене доменных служб Azure Active Directory (Azure AD DS), можно использовать службы удаленных рабочих столов (RDS) и сервер политики сети (NPS). Azure AD DS выполняет проверку подлинности пользователей при запросе доступа через среду RDS. Для повышения безопасности можно интегрировать многофакторную идентификацию Azure AD, чтобы предоставить дополнительный запрос проверки подлинности во время событий входа. Многофакторная идентификация Azure AD использует расширение для NPS для предоставления этой функции.

> [!IMPORTANT]
> Рекомендуемый способ безопасного подключения к виртуальным машинам в управляемом домене AD DS Azure — это использование Azure бастиона, полностью управляемой платформой службы PaaS, которую вы предоставляете в виртуальной сети. Узел бастиона обеспечивает безопасное и простое подключение протокол удаленного рабочего стола (RDP) к виртуальным машинам непосредственно в портал Azure по протоколу SSL. При подключении через узел бастиона виртуальным машинам не требуется общедоступный IP-адрес, и нет необходимости использовать группы безопасности сети для предоставления доступа к протоколу удаленного рабочего стола через TCP-порт 3389.
>
> Мы настоятельно рекомендуем использовать Azure бастиона во всех регионах, где она поддерживается. В регионах без доступности Azure бастиона выполните действия, описанные в этой статье, пока не будет доступна служба Azure бастиона. Соблюдайте осторожность при назначении общедоступных IP-адресов виртуальным машинам, присоединенным к Azure AD DS, где разрешен весь входящий трафик RDP.
>
> Дополнительные сведения см. в статье [что такое Azure бастиона?][bastion-overview].

В этой статье показано, как настроить RDS в Azure AD DS и при необходимости использовать расширение NPS многофакторной идентификации Azure AD.

## <a name="prerequisites"></a>Предварительные требования

Для работы с этой статьей требуются следующие ресурсы:

* Активная подписка Azure.
    * Если у вас еще нет подписки Azure, [создайте учетную запись](https://azure.microsoft.com/free/?WT.mc_id=A261C142F).
* Связанный с вашей подпиской клиент Azure Active Directory, синхронизированный с локальным или облачным каталогом.
    * Если потребуется, [создайте клиент Azure Active Directory][create-azure-ad-tenant] или [свяжите подписку Azure со своей учетной записью][associate-azure-ad-tenant].
* Управляемый домен доменных служб Azure Active Directory, включенный и настроенный в клиенте Azure AD.
    * При необходимости [создайте и настройте управляемый домен доменных служб Azure Active Directory][create-azure-ad-ds-instance].
* Подсеть *рабочих нагрузок* , созданная в виртуальной сети Azure Active Directory доменных служб.
    * При необходимости [настройте виртуальную сеть для управляемого домена доменных служб Azure Active Directory][configure-azureadds-vnet].
* Учетная запись пользователя, входящая в группу *администраторов Azure AD DC* в клиенте Azure AD.

## <a name="deploy-and-configure-the-remote-desktop-environment"></a>Развертывание и настройка среды удаленный рабочий стол

Чтобы приступить к работе, создайте как минимум две виртуальные машины Azure под управлением Windows Server 2016 или Windows Server 2019. Для обеспечения избыточности и высокой доступности среды удаленный рабочий стол (RD) можно позднее добавить и сбалансировать нагрузку дополнительных узлов.

Предлагаемое развертывание RDS включает следующие две виртуальные машины:

* *RDGVM01* — запускает сервер RDCB, сервер удаленных рабочих столов веб-доступ и сервер шлюза удаленных рабочих столов.
* *RDSHVM01* — запускает сервер узла сеансов удаленных рабочих столов.

Убедитесь, что виртуальные машины развернуты в подсети *рабочей нагрузки* виртуальной сети Azure AD DS, а затем присоедините виртуальные машины к управляемому домену. Дополнительные сведения см. в статье как [создать виртуальную машину Windows Server и присоединить ее к управляемому домену][tutorial-create-join-vm].

Развертывание среды удаленных рабочих столов содержит ряд шагов. Существующее Руководство по развертыванию удаленных рабочих столов можно использовать без каких-либо специфических изменений для использования в управляемом домене.

1. Войдите на виртуальные машины, созданные для среды удаленных рабочих столов, используя учетную запись, которая является частью группы *администраторов контроллера домена Azure AD* , например *contosoadmin*.
1. Чтобы создать и настроить RDS, используйте существующее [Удаленный рабочий столное руководством по развертыванию среды][deploy-remote-desktop]. При необходимости распространите компоненты сервера удаленных рабочих столов между виртуальными машинами Azure.
    * Относится к Azure AD DS. при настройке лицензирования удаленных рабочих столов задайте для него режим " **на устройство** ", а не для **каждого пользователя** , как указано в разделе "рекомендации по развертыванию".
1. Если вы хотите предоставить доступ с помощью веб-браузера, [Настройте веб-клиент удаленный рабочий стол для пользователей][rd-web-client].

С помощью удаленного рабочего стола, развернутого в управляемом домене, вы можете управлять службой и использовать ее так же, как с локальным доменом AD DS.

## <a name="deploy-and-configure-nps-and-the-azure-ad-mfa-nps-extension"></a>Развертывание и настройка NPS и расширения NPS для Azure AD MFA

Если требуется повысить уровень безопасности при входе пользователя в систему, можно дополнительно интегрировать среду удаленных рабочих столов с многофакторной идентификацией Azure AD. В этой конфигурации пользователи получают дополнительный запрос во время входа, чтобы подтвердить свою личность.

Для обеспечения этой возможности в среде устанавливается дополнительный сервер политики сети (NPS) вместе с расширением NPS многофакторной идентификации Azure AD. Это расширение интегрируется с Azure AD для запроса и возврата состояния запросов многофакторной проверки подлинности.

Пользователи должны быть [зарегистрированы для использования многофакторной идентификации Azure AD][user-mfa-registration], которая может потребовать дополнительных лицензий Azure AD.

Чтобы интегрировать многофакторную идентификацию Azure AD в среду Azure AD DS удаленный рабочий стол, создайте NPS-сервер и установите расширение.

1. Создайте дополнительную виртуальную машину Windows Server 2016 или 2019, например *NPSVM01*, которая подключена к подсети *рабочих нагрузок* в виртуальной сети Azure AD DS. Присоедините виртуальную машину к управляемому домену.
1. Войдите на виртуальную машину NPS в качестве учетной записи, которая является частью группы *администраторов контроллера домена Azure AD* , например *contosoadmin*.
1. В **Диспетчер сервера** выберите **Добавить роли и компоненты**, а затем установите роль *службы политики сети и доступа* .
1. Используйте существующую статью с инструкциями по [установке и настройке расширения NPS для Azure AD MFA][nps-extension].

После установки расширения NPS для NPS-сервера и Azure AD многофакторной идентификации выполните следующий раздел, чтобы настроить его для использования с средой удаленных рабочих столов.

## <a name="integrate-remote-desktop-gateway-and-azure-ad-multi-factor-authentication"></a>Интеграция шлюза удаленный рабочий стол и службы многофакторной идентификации Azure AD

Чтобы интегрировать расширение NPS для многофакторной идентификации Azure AD, используйте существующую статью с инструкциями по [интеграции инфраструктуры шлюза удаленный рабочий стол с помощью расширения сервера политики сети (NPS) и Azure AD][azure-mfa-nps-integration].

Для интеграции с управляемым доменом необходимо выполнить следующие дополнительные параметры конфигурации:

1. Не [регистрировать сервер NPS в Active Directory][register-nps-ad]. Этот шаг завершается ошибкой в управляемом домене.
1. На [шаге 4, чтобы настроить политику сети][create-nps-policy], также установите флажок **игнорировать свойства удаленного доступа учетной записи пользователя**.
1. Если вы используете Windows Server 2019 для расширения NPS сервера NPS и многофакторной идентификации Azure AD, выполните следующую команду, чтобы обновить безопасный канал, чтобы разрешить серверу NPS правильно обмениваться данными:

    ```powershell
    sc sidtype IAS unrestricted
    ```

Теперь пользователям предлагается дополнительный фактор проверки подлинности при входе в систему, например текстовое сообщение или запрос в Microsoft Authenticator приложении.

# <a name="azure-active-directory-domain-services-deployment-and-management-for-azure-cloud-solution-providers"></a>Azure Active Directory развертывания и управления доменными службами для поставщиков облачных решений Azure

Поставщики облачных решений Azure (CSP) — это программа для партнеров Майкрософт, которая предоставляет канал лицензии для различных облачных служб Майкрософт. Azure CSP дает партнерам возможность управлять продажами и выставлением счетов, обеспечивать поддержку по выставлению счетов и техническую поддержку и быть единственным непосредственным контактом клиентов. Кроме того, Azure CSP предоставляет полный набор инструментов, включая портал самообслуживания и сопутствующие интерфейсы API. Эти инструменты позволяют партнерам CSP легко подготавливать ресурсы Azure и управлять ими, а также выставлять счета для клиентов и их подписок.

[Портал центра партнеров](/partner-center/azure-plan-lp) является точкой входа для всех партнеров Azure CSP и предоставляет широкие возможности управления клиентами, автоматическую обработку и многое другое. Партнеры Azure CSP могут использовать возможности Центра партнеров с помощью пользовательского веб-интерфейса или с помощью PowerShell и различных вызовов API.

На приведенной ниже схеме показан общий принцип работы модели CSP. Здесь у Contoso есть клиент Azure Active Directory (Azure AD). Она является партнером CSP, и служба Azure CSP развертывает ресурсы в ее подписке Azure CSP и управляет ими. У компании Contoso также могут быть обычные (прямые) подписки Azure, счета за которые выставляются непосредственно этой компании.

![Общие сведения о модели CSP](./media/csp/csp_model_overview.png)

Клиент партнера CSP имеет три специальные группы агентов — агенты *администратора* , агенты службы *поддержки* и агенты *продаж* .

Группе *агенты администратора* назначается роль администратора клиента в клиенте Azure AD contoso. В результате пользователь, входящий в группу агентов администрирования партнера CSP, имеет права администратора клиента в клиенте Azure AD contoso.

Когда партнер CSP подготавливает подписку Azure CSP для компании Contoso, их группе агентов администрирования назначается роль владельца для этой подписки. В результате агенты администрирования партнера CSP получают необходимые привилегии для подготовки ресурсов Azure, таких как виртуальные машины, виртуальные сети и доменные службы Azure AD, от имени компании Contoso.

Дополнительные сведения см. в [обзоре Azure CSP](/partner-center/azure-plan-lp).

## <a name="benefits-of-using-azure-ad-ds-in-an-azure-csp-subscription"></a>Преимущества использования AD DS Azure в подписке Azure CSP

Azure Active Directory доменных служб (Azure AD DS) предоставляет управляемые доменные службы, такие как присоединение к домену, групповая политика, LDAP, проверка подлинности Kerberos/NTLM, полностью совместимая с Windows Server домен Active Directory Services. На протяжении десятилетий было создано множество приложений для работы с AD и использования этих возможностей. Многие независимые поставщики программного обеспечения создавали и развертывали приложения в локальной среде клиентов. Эти приложения трудно поддерживать, так как часто требуется доступ к различным средам, в которых развертываются приложения. Благодаря подпискам Azure CSP вы получаете более простую альтернативу с возможностями масштабирования и гибкости Azure.

Azure AD DS поддерживает подписки Azure CSP. Вы можете развернуть приложение в подписке Azure CSP, привязанной к клиенту Azure AD клиента. В результате сотрудники (сотрудники службы поддержки) могут управлять виртуальными машинами, на которых развернуто приложение, и обслуживать их, используя корпоративные учетные данные Организации.

Вы также можете развернуть управляемый домен AD DS Azure в клиенте Azure AD клиента. Затем ваше приложение подключается к управляемому домену клиента. Возможности приложения, которые используют протоколы Kerberos/NTLM, LDAP или [API System. DirectoryServices](/dotnet/api/system.directoryservices) , беспрепятственно работают с доменом вашего клиента. Клиенты могут использовать приложение как услугу, не заботясь о поддержке инфраструктуры, в которой развертывается приложение.

Все выставление счетов за ресурсы Azure, используемые в этой подписке, включая Azure AD DS, взимается обратно. Вы сохраняете полный контроль над связью с клиентом, когда дело доходит до продаж, выставления счетов, технической поддержки и т. д. Благодаря гибкости платформы Azure CSP небольшая группа агентов поддержки может обслуживать множество клиентов, которые развернули экземпляры вашего приложения.

## <a name="csp-deployment-models-for-azure-ad-ds"></a>Модели развертывания CSP для Azure AD DS

Вы можете использовать Azure AD DS с подпиской Azure CSP двумя способами. Выберите соответствующий способ, приняв во внимание требования к безопасности и простоте клиентов.

### <a name="direct-deployment-model"></a>Модель прямого развертывания

В этой модели развертывания AD DS Azure включена в виртуальной сети, которая принадлежит к подписке Azure CSP. Агенты администрирования партнера CSP имеют следующие привилегии:

* Права *глобального администратора* в клиенте Azure AD клиента.
* Права *владельца подписки* на подписку Azure CSP.

![Модель прямого развертывания](./media/csp/csp_direct_deployment_model.png)

В этой модели развертывания агенты администрирования поставщика CSP могут администрировать удостоверения клиента. Эти агенты администрирования могут выполнять такие задачи, как подготавливать новых пользователей или группы, а также добавлять приложения в клиент Azure AD клиента.

Эта модель развертывания может быть оптимальной для небольших организаций, у которых нет выделенного администратора идентификации, или вы предпочитаете партнеру CSP администрировать удостоверения от их имени.

### <a name="peered-deployment-model"></a>Пиринговая модель развертывания

В этой модели развертывания AD DS Azure включена в виртуальной сети, принадлежащей клиенту — прямую подписку Azure, уплаченную клиентом. Партнер CSP может развертывать приложения в виртуальной сети, принадлежащей подписке поставщика CSP клиента. Эти виртуальные сети можно подключить с помощью пиринга виртуальных сетей Azure.

В этом развертывании рабочие нагрузки или приложения, развернутые партнером CSP в подписке Azure CSP, могут подключаться к управляемому домену клиента, подготовленному в непосредственной подписке Azure клиента.

![Пиринговая модель развертывания](./media/csp/csp_peered_deployment_model.png)

Такая модель развертывания обеспечивает разделение привилегий и позволяет агентам службы технической поддержки партнера CSP администрировать подписку Azure, а также развертывать в ней ресурсы и управлять ими. Однако агентам службы поддержки партнера CSP не требуется иметь права глобального администратора в каталоге Azure AD клиента. Администраторы удостоверений клиента могут по-прежнему управлять удостоверениями в своей организации.

Эта модель развертывания может быть оптимальной для сценариев, в которых ISV предоставляет размещенную версию своего локального приложения, которая также должна подключаться к Azure AD клиента.

## <a name="administer-azure-ad-ds-in-csp-subscriptions"></a>Администрирование AD DS Azure в подписках CSP

Ниже приведены важные рекомендации по администрированию управляемого домена в подписке Azure CSP.

* **Агенты администрирования CSP могут подготавливать управляемый домен с помощью своих учетных данных:** Azure AD DS поддерживает подписки Azure CSP. Пользователи, принадлежащие к группе агентов администрирования партнера CSP, могут подготавливать новый управляемый домен.

* **Поставщики служб шифрования могут создавать сценарии создания новых управляемых доменов для своих клиентов с помощью PowerShell:** Дополнительные сведения см. [в статье включение AD DS Azure с помощью PowerShell](powershell-create-instance.md) .

* **Агенты администрирования CSP не могут выполнять текущие задачи управления в управляемом домене с использованием их учетных данных:** Пользователи с правами администратора CSP не могут выполнять стандартные задачи управления в управляемом домене, используя свои учетные данные. Эти пользователи являются внешними по отношению к клиенту Azure AD клиента, и их учетные данные недоступны в клиенте Azure AD клиента. Azure AD DS не имеет доступа к хэшам паролей Kerberos и NTLM для этих пользователей, поэтому пользователи не могут пройти проверку подлинности в управляемых доменах.

  > [!WARNING]
  > Необходимо создать учетную запись пользователя в каталоге клиента для выполнения повседневных задач администрирования в управляемом домене.
  >
  > Вы не можете войти в управляемый домен, используя учетные данные пользователя администратора CSP. Для этого используйте учетные данные учетной записи пользователя, принадлежащей клиенту Azure AD клиента. Эти учетные данные необходимы для таких задач, как присоединение виртуальных машин к управляемому домену, администрирование DNS или администрирование групповая политика.

* **Учетная запись пользователя, созданная для постоянного администрирования, должна быть добавлена в группу *администраторов контроллера домена AAD* :** группа *администраторов контроллера домена AAD* имеет права на выполнение определенных делегированных задач администрирования в управляемом домене. Эти задачи включают настройку DNS, создание подразделений и администрирование групповой политики.
    
    Чтобы партнер CSP выполнял эти задачи в управляемом домене, необходимо создать учетную запись пользователя в клиенте Azure AD клиента. Учетные данные для этой учетной записи должны совместно использоваться агентами администрирования партнера CSP. Кроме того, эту учетную запись пользователя необходимо добавить в группу *администраторов контроллера домена AAD* , чтобы разрешить выполнение задач настройки в управляемом домене с использованием этой учетной записи пользователя.

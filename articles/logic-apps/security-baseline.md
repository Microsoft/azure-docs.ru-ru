# <a name="azure-security-baseline-for-logic-apps"></a>Базовый план безопасности Azure для Logic Apps

Базовый план безопасности Azure для Logic Apps содержит рекомендации, которые помогут повысить уровень безопасности развертывания.

Базовый план безопасности для этой службы взят из [Эталона безопасности Azure версии 1.0](../security/benchmarks/overview.md), содержащего рекомендации по обеспечению безопасности облачных решений в Azure с помощью наших практических рекомендаций.

Дополнительные сведения см. в статье [Обзор базовой конфигурации безопасности Azure](../security/benchmarks/security-baselines-overview.md).

### <a name="11-protect-azure-resources-within-virtual-networks"></a>1,1: защита ресурсов Azure в виртуальных сетях

**Рекомендации**. соединители, выполняемые в глобальной службе Logic Apps с несколькими клиентами, развертываются и управляются корпорацией Майкрософт. Эти соединители предоставляют триггеры и действия для доступа к облачным службам, локальным системам или обоим, включая Office 365, хранилище BLOB-объектов Azure, SQL Server, Dynamics, Salesforce, SharePoint и т. д. Вы можете использовать тег службы Азуреконнекторс при указании правил в группе безопасности сети или брандмауэре Azure, чтобы разрешить доступ к связанным ресурсам.

Для приложений логики, которым необходим прямой доступ к ресурсам в виртуальной сети Azure, можно создать среду службы интеграции (ISE), в которой можно создавать, развертывать и запускать приложения логики на выделенных ресурсах. Некоторые виртуальные сети Azure используют частные конечные точки (частные ссылки Azure) для предоставления доступа к службам Azure PaaS, таким как служба хранилища Azure, Azure Cosmos DB, база данных SQL Azure, партнерские службы или службы клиентов, размещенные в Azure. Если приложениям логики требуется доступ к виртуальным сетям, использующим частные конечные точки, необходимо создать, развернуть и запустить эти приложения логики в интегрированной среде сценариев.

При создании интегрированной среды сценариев можно выбрать использование внутренних или внешних конечных точек доступа. Выбор определяет, могут ли триггеры запроса или веб-перехватчика в интегрированной среде сценариев получать вызовы извне вашей виртуальной сети. Внутренние и внешние конечные точки доступа также влияют на то, можно ли просматривать журнал выполнения приложения логики, включая входные и выходные данные для запуска внутри или за пределами виртуальной сети.

Убедитесь, что все развертывания подсетей виртуальных сетей, связанные с ИНТЕГРИРОВАНной средой, имеют группу безопасности сети, применяемую к элементам управления доступом к сети, относящимся к доверенным портам и источникам приложения. При развертывании приложений логики в интегрированной среде сценариев используйте ссылку Private. Частная ссылка Azure позволяет получить доступ к службам Azure PaaS и службам, размещенным в Azure, с помощью частной конечной точки в виртуальной сети. Кроме того, если у вас есть конкретный вариант использования, вы можете удовлетворить это требование, реализовав брандмауэр Azure. Чтобы сократить сложность при настройке правил безопасности, используйте теги служб, представляющие группы префиксов IP-адресов для конкретной службы Azure.

### <a name="14-deny-communications-with-known-malicious-ip-addresses"></a>1,4: запретите обмен данными с известными вредоносными IP-адресами

**Руководство**. Если приложение логики использует триггер на основе запроса, который получает входящие вызовы или запросы, такие как запрос или триггер веб-перехватчика, можно ограничить доступ, чтобы только полномочные клиенты могли вызывать приложение логики.

Если вы используете приложения логики в среде службы интеграции (ISE), включите стандарт защиты от атак DDoS в виртуальной сети, связанной с ИНТЕГРИРОВАНной средой, чтобы защититься от атак от атак DDoS. Используйте интегрированную аналитику угроз Центра безопасности Azure, чтобы запретить обмен данными с известными вредоносными или неиспользуемыми IP-адресами Интернета.

Разверните брандмауэр Azure на всех границах сети Организации с включенной логикой "предупреждения и отказ" для вредоносного сетевого трафика.

Используйте центр безопасности Azure JIT-доступ к сети, чтобы настроить группы безопасности сети, чтобы ограничить раскрытие конечных точек с утвержденными IP адресами в течение ограниченного периода времени.

Используйте адаптивную защиту сети в центре безопасности Azure, чтобы рекомендовать NSG конфигурации, ограничивающие порты и исходные IP-адреса на основе фактического трафика и аналитики угроз.

# <a name="questions-about-backing-up-azure-files"></a>Вопросы о резервном копировании файлов Azure

В этой статье содержатся ответы на часто задаваемые вопросы о резервном копировании файлов Azure. В некоторых ответах приведены ссылки на статьи, содержащие более подробные сведения. Кроме того, вопросы о службе Azure Backup можно задать на [странице вопросов и ответов на сайте Майкрософт](/answers/topics/azure-backup.html).

Чтобы быстро просмотреть содержание этой статьи, используйте ссылки справа в разделе **В этой статье**.

## <a name="configuring-the-backup-job-for-azure-files"></a>Настройка задания резервного копирования файлов Azure

### <a name="why-cant-i-see-some-of-my-storage-accounts-that-i-want-to-protect-which-contain-valid-azure-file-shares"></a>Почему я не вижу некоторые учетные записи хранения, содержащие общие папки, которые требуется защитить в Azure?

Чтобы проверить, относится ли учетная запись хранения к одному из поддерживаемых типов учетных записей хранения, см. [таблицу поддержки резервного копирования общих папок в Azure](azure-file-share-support-matrix.md). Также вполне возможно, что учетная запись хранения, которую вы ищете, уже защищена или зарегистрирована в другом хранилище. [Отмените регистрацию учетной записи хранения](manage-afs-backup.md#unregister-a-storage-account) в хранилище, чтобы найти учетную запись хранения, которую требуется защитить, в других хранилищах.

### <a name="why-cant-i-see-some-of-my-azure-file-shares-in-the-storage-account-when-im-trying-to-configure-backup"></a>Почему я не вижу некоторые файловые ресурсы Azure в учетной записи хранения при попытке настроить резервное копирование?

Проверьте, не защищен ли файловый ресурс Azure в том же хранилище служб восстановления и не удален ли он.

### <a name="can-i-protect-file-shares-connected-to-a-sync-group-in-azure-files-sync"></a>Могу ли я защитить файловые ресурсы, подключенные к группе синхронизации в службе синхронизации файлов Azure?

Да. Функция защиты общих папок Azure, подключенных к группам синхронизации, включена.

### <a name="when-trying-to-back-up-file-shares-i-selected-a-storage-account-to-discover-the-file-shares-in-it-however-i-didnt-protect-them-how-do-i-protect-these-file-shares-with-any-other-vault"></a>При попытке резервного копирования файловых ресурсов я выбрал учетную запись хранения для обнаружения общих файловых ресурсов. Но мне не удалось защитить эти ресурсы. Как защитить эти общие папки с помощью другого хранилища?

Если во время создания резервной копии выбрать учетную запись хранения для обнаружения общих папок в ней, учетная запись хранения регистрируется в хранилище, из которого выбирается учетная запись. Если вы выбираете для защиты общие папки в другом хранилище, [отмените регистрацию](manage-afs-backup.md#unregister-a-storage-account) выбранной учетной записи хранения в этом хранилище.

### <a name="why-cant-i-change-the-vault-to-configure-backup-for-the-file-share"></a>Почему не удается изменить хранилище для настройки резервного копирования для файлового ресурса?

Если учетная запись хранения уже зарегистрирована в хранилище или другие файловые ресурсы в учетной записи хранения защищены с помощью хранилища, вы не можете изменить его. Все общие файловые ресурсы в учетной записи хранения могут быть защищены только тем же хранилищем. Если вы хотите изменить хранилище, необходимо [Отключить защиту для всех файловых ресурсов в учетной записи хранения](manage-afs-backup.md#stop-protection-on-a-file-share) из подключенного хранилища, [отменить регистрацию](manage-afs-backup.md#unregister-a-storage-account) учетной записи хранения, а затем выбрать другое хранилище для защиты.

### <a name="can-i-change-the-vault-to-which-i-back-up-my-file-shares"></a>Можно ли изменить хранилище, в котором создана резервная копия моих файловых ресурсов?

Да. Но вам потребуется [остановить защиту общей папки](manage-afs-backup.md#stop-protection-on-a-file-share) из подключенного хранилища, [отменить регистрацию](manage-afs-backup.md#unregister-a-storage-account) этой учетной записи хранения и защитить ее из другого хранилища.

### <a name="can-i-protect-two-different-file-shares-from-the-same-storage-account-to-different-vaults"></a>Можно ли создать резервные копии двух разных файловых ресурсов из одной учетной записи хранения в разных хранилищах?

Нет. Все файловые ресурсы в учетной записи хранения могут быть защищены только одним хранилищем.

## <a name="backup"></a>Резервное копирование

### <a name="what-should-i-do-if-my-backups-start-failing-due-to-the-maximum-limit-reached-error"></a>Что делать, если операции резервного копирования завершаются сбоем из-за достижения максимального допустимого количества моментальных снимков?

В любое время вы можете хранить до 200 моментальных снимков для файлового ресурса. При этом учитываются моментальные снимки, сделанные службой Azure Backup в соответствии с политикой. Если операции резервного копирования завершаются сбоем после достижения этого ограничения, удалите моментальные снимки по запросу.

## <a name="restore"></a>Восстановить

### <a name="can-i-recover-from-a-deleted-azure-file-share"></a>Можно ли выполнить восстановление из удаленного файлового ресурса Azure?

Если общая папка находится в состоянии обратимого удаления, необходимо сначала отменить удаление общей папки для выполнения операции восстановления. Операция отмены удаления переместит общую папку в активное состояние, где можно выполнить восстановление в любой момент времени. Чтобы узнать, как отменить удаление файлового ресурса, перейдите по [этой ссылке](../storage/files/storage-files-enable-soft-delete.md?tabs=azure-portal#restore-soft-deleted-file-share) или просмотрите [сценарий отмены удаления файлового ресурса](./scripts/backup-powershell-script-undelete-file-share.md). Если файловый ресурс окончательно удаляется, вы не сможете восстановить содержимое и моментальные снимки.

### <a name="can-i-restore-from-backups-if-i-stopped-protection-on-an-azure-file-share"></a>Можно ли выполнить восстановление из резервных копий после отключения защиты в файловом ресурсе Azure?

Да. Если вы выбрали **Retain Backup Data** (Сохранить данные архивации) при отключении защиты, то можете выполнить восстановление из всех имеющихся точек.

### <a name="what-happens-if-i-cancel-an-ongoing-restore-job"></a>Что произойдет, если отменить выполняемое задание восстановления?

Если отменить выполняемое задание восстановления, процесс восстановления прекращается и все файлы, восстановленные до отмены, остаются в заданном целевом расположении (исходном или другом указанном) без выполнения отката.

## <a name="manage-backup"></a>Управление резервным копированием

### <a name="can-i-use-powershell-to-configuremanagerestore-backups-of-azure-file-shares"></a>Можно ли использовать PowerShell для настройки, администрирования и восстановления резервных копий файловых ресурсов Azure?

Да. Дополнительные сведения см. [здесь](backup-azure-afs-automation.md).

### <a name="can-i-access-the-snapshots-taken-by-azure-backups-and-mount-them"></a>Можно ли получить доступ к моментальным снимкам, сделанным службой Azure Backup, и подключить их?

Все моментальные снимки, сделанные службой Azure Backup, можно просмотреть на портале, с помощью PowerShell или интерфейса командной строки. Дополнительные сведения о моментальных снимках общих папок службы "Файлы Azure" см. в статье [Общие сведения о моментальных снимках общих ресурсов для службы файлов Azure](../storage/files/storage-snapshots-files.md).

### <a name="what-happens-after-i-move-a-backed-up-file-share-to-a-different-subscription"></a>Что происходит после перемещения резервной копии файлового ресурса в другую подписку?

После перемещения общей папки в другую подписку она считается новым файловым ресурсом Azure Backup. Ниже приведены рекомендуемые действия.
 
Сценарий. Предположим, что у вас есть файловый ресурс *FS1* в подписке *S1* и он защищен с помощью хранилища *v1* . Теперь вы хотите переместить файловый ресурс в подписку *S2*.
 
1.  Переместите нужную учетную запись хранения и общую папку (FS1) в другую подписку (S2).
2.  В хранилище v1 активируйте операцию отключения защиты с помощью операции удаления данных для FS1.
3.  Отмените регистрацию учетной записи хранения, в которой размещается FS1, из хранилища v1.
4.  Перенастройте резервное копирование для FS1, теперь перемещенное на S2, с хранилищем (v2) в подписке S2. 
 
Обратите внимание, что после повторной настройки резервного копирования с версии 2 моментальные снимки, сделанные с версией 1, больше не будут управляться с помощью Azure Backup. Поэтому необходимо удалить эти моментальные снимки вручную в соответствии с вашими требованиями.

### <a name="can-i-move-my-backed-up-file-share-to-a-different-resource-group"></a>Можно ли переместить резервную копию файлового ресурса в другую группу ресурсов?
 
Да, вы можете переместить резервную копию файлового ресурса в другую группу ресурсов. Тем не менее, необходимо будет повторно настроить резервное копирование для общей папки, так как это Azure Backup. Кроме того, моментальные снимки, созданные до перемещения группы ресурсов, больше не будут управляться службой Azure Backup. Поэтому необходимо удалить эти моментальные снимки вручную в соответствии с вашими требованиями.

### <a name="what-is-the-maximum-retention-i-can-configure-for-backups"></a>Какой максимальный срок хранения можно настроить для резервных копий?

Сведения о максимальном сроке хранения см. в [таблице поддержки](azure-file-share-support-matrix.md). Azure Backup выполняет вычисление количества моментальных снимков в реальном времени при вводе значений срока хранения во время настройки политики резервного копирования. Как только число моментальных снимков, соответствующих заданным значениям срока хранения, превысит 200, на портале отобразится предупреждение с запросом на изменение значений срока хранения. Это необходимо для того, чтобы в какой-либо момент времени не было превышено максимальное количество моментальных снимков, поддерживаемое службой "Файлы Azure" для общей папки.

### <a name="what-is-the-impact-on-existing-recovery-points-and-snapshots-when-i-modify-the-backup-policy-for-an-azure-file-share-to-switch-from-daily-policy-to-gfs-policy"></a>Что будет с существующими точками восстановления и моментальными снимками, если я изменю политику резервного копирования для общей папки Azure, переключившись с политики ежедневного резервного копирования на политику GFS?

При изменении политики ежедневного резервного копирования на политику GFS, в которой добавляется еженедельное, ежемесячное и ежегодное резервное копирование, происходит следующее.

- **Период хранения.** Если при изменении политики вы добавляете еженедельное, ежемесячное и ежегодное хранение, все будущие точки восстановления, созданные в рамках запланированного резервного копирования, будут помечены в соответствии с новой политикой. Все существующие точки восстановления по-прежнему будут рассматриваться как ежедневные точки восстановления, поэтому они не будут помечены как еженедельные, ежемесячные или ежегодные.

- **Очистка моментальных снимков и точек восстановления.**

  - При продлении ежедневного хранения дата окончания срока действия существующих точек восстановления обновляется в соответствии со значением срока хранения для ежедневной резервной копии, настроенным в новой политике.
  - При уменьшении срока хранения существующие точки восстановления и моментальные снимки будут помечены для удаления в следующем задании очистки согласно значению срока хранения, настроенному в новой политике, а затем удалены.

Ниже приведен пример процесса.

#### <a name="existing-policy-p1"></a>Существующая политика [P1]

|Тип хранения |Расписание |Сохранение  |
|---------|---------|---------|
|Ежедневно    |    Каждый день в 20:00    |  100 дней       |

#### <a name="new-policy-modified-p1"></a>Новая политика [измененная P1]

| Тип хранения | Расписание                       | Сохранение |
| -------------- | ------------------------------ | --------- |
| Ежедневно          | Каждый день в 21:00              | 50 дней   |
| Weekly (Еженедельно);         | По воскресеньям в 21:00              | 3 недели   |
| Ежемесячная        | В последний понедельник месяца в 21:00         | 1 месяц   |
| Ежегодная         | В третье воскресенье января в 21:00 | 4 года   |

#### <a name="impact"></a>Влияние

1. Дата окончания срока действия существующих ежедневных точек восстановления будет скорректирована в соответствии с установленным новой политикой сроком хранения ежедневной резервной копии, который составляет 50 дней. Точки восстановления старше 50 дней будут помечены для удаления.

2. Существующие точки восстановления не будут помечены как еженедельные, ежемесячные или ежегодные на основе новой политики.

3. Резервное копирование будет запускаться в соответствии с новым расписанием, то есть в 21:00.

4. Срок действия всех будущих точек восстановления будет согласован с новой политикой.

>[!NOTE]
>Изменения политики будут влиять только на точки восстановления, созданные в рамках запланированного запуска задания резервного копирования. Для резервного копирования по запросу срок хранения определяется значением параметра **Retain Till** (Срок хранения резервной копии), указанного во время создания резервной копии.

### <a name="what-is-the-impact-on-existing-recovery-points-when-i-modify-an-existing-gfs-policy"></a>Как влияет на существующие точки восстановления изменение существующей политики GFS?

При применении новой политики к общим папкам все будущие запланированные резервные копии будут записываться в соответствии с расписанием в измененной политике.  Срок хранения всех существующих точек восстановления изменяется в соответствии с новыми значениями срока хранения. Если срок хранения увеличивается, существующие точки восстановления помечаются для хранения в соответствии с новой политикой. Если период хранения сокращается, они помечаются для очистки при выполнении следующего задания очистки, а затем удаляются.

Ниже приведен пример процесса.

#### <a name="existing-policy-p2"></a>Существующая политика [P2]

| Тип хранения | Расписание           | Сохранение |
| -------------- | ------------------ | --------- |
| Ежедневно          | Каждый день в 20:00 | 50 дней   |
| Weekly (Еженедельно);         | По понедельникам в 20:00  | 3 недели   |

#### <a name="new-policy-modified-p2"></a>Новая политика [измененная P2]

| Тип хранения | Расписание               | Сохранение |
| -------------- | ---------------------- | --------- |
| Ежедневно          | Каждый день в 21:00     | 10 дней   |
| Weekly (Еженедельно);         | По понедельникам в 21:00      | 2 недели   |
| Ежемесячная        | В последний понедельник месяца в 21:00 | 2 месяца  |

#### <a name="impact-of-change"></a>Влияние изменений

1. Дата окончания срока действия существующих ежедневных точек восстановления будет изменена в соответствии с новым сроком хранения ежедневной резервной копии, который составляет 10 дней. Поэтому все ежедневные точки восстановления старше 10 дней будут удалены.

2. Дата окончания срока действия существующих еженедельных точек восстановления будет изменена в соответствии с новым сроком хранения еженедельной резервной копии, который составляет две недели. Поэтому все еженедельные точки восстановления старше двух недель будут удалены.

3. Ежемесячные точки восстановления будут создаваться только в рамках будущих резервных копий на основе новой конфигурации политики.

4. Срок действия всех будущих точек восстановления будет согласован с новой политикой.

>[!NOTE]
>Изменения политики будут влиять только на точки восстановления, созданные в рамках запланированного резервного копирования. Для резервного копирования по запросу срок хранения определяется значением параметра **Retain Till** (Срок хранения резервной копии), указанного во время создания резервной копии.

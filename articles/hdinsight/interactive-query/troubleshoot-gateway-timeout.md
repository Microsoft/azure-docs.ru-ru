# <a name="exception-when-running-queries-from-apache-ambari-hive-view-in-azure-hdinsight"></a>Исключение при выполнении запросов из представления Hive Apache Ambari в Azure HDInsight

В этой статье описываются действия по устранению неполадок и возможные способы решения проблем при взаимодействии с кластерами Azure HDInsight.

## <a name="issue"></a>Проблема

При выполнении Apache Hive запроса из представления Hive Apache Ambari, периодически появляется следующее сообщение об ошибке:

```error
Cannot create property 'errors' on string '<!DOCTYPE html PUBLIC '-//W3C//DTD XHTML 1.0 Strict//EN' 'http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd'>
<html xmlns='http://www.w3.org/1999/xhtml'>
<head>
<title>IIS 8.5 Detailed Error - 502.3 - Bad Gateway</title>
```

## <a name="cause"></a>Причина

Время ожидания шлюза.

Значение времени ожидания шлюза составляет 2 минуты. Запросы из представления Hive Ambari передаются в `/hive2` конечную точку через шлюз. После успешной компиляции и принятия запроса HiveServer возвращает `queryid` . Затем клиенты продолжают опрашивать состояние запроса. В ходе этого процесса, если HiveServer не возвращает HTTP-ответ в течение 2 минут, шлюз HDI выдает вызывающему объекту ошибку времени ожидания шлюза 502,3. Ошибки могут возникать при отправке запроса на обработку (более вероятно), а также в вызове get Status (менее вероятно). Пользователи могут видеть любой из них.

Поток обработчика HTTP должен быть быстрым: подготовка задания и возврат `queryid` . Однако из-за нескольких причин все потоки обработчиков могут быть заняты, что приведет к превышению времени ожидания для новых запросов и вызовам Get Status.

### <a name="responsibilities-of-the-http-handler-thread"></a>Обязанности потока обработчика HTTP

Когда клиент отправляет запрос в HiveServer, он выполняет в основном потоке следующие действия:

* Выполнить синтаксический анализ запроса, выполнить семантическую проверку
* Получить блокировку
* Хранилище метаданных Поиск при необходимости
* Компиляция запроса (DDL или DML)
* Подготовка плана запроса
* Выполнение авторизации (запуск всех применимых политик Ranger в защищенных кластерах)

## <a name="resolution"></a>Решение

Некоторые общие рекомендации по улучшению ситуации:

* Если используется внешний хранилище метаданных Hive, проверьте метрики БД и убедитесь, что база данных не перегружена. Рассмотрите возможность масштабирования уровня базы данных хранилище метаданных.

* Убедитесь, что параллельные Ops включены (это позволяет потокам обработчика HTTP выполняться параллельно). Чтобы проверить это значение, запустите [Apache Ambari](../hdinsight-hadoop-manage-ambari.md) и **перейдите к**  >    >  **расширенному**  >  **пользовательскому кусту** Hive. Значение для `hive.server2.parallel.ops.in.session` должно быть `true` .

* Убедитесь, что номер SKU виртуальной машины кластера слишком мал для загрузки. Рассмотрите возможность разделения работы между несколькими кластерами. Дополнительные сведения см. [в разделе Выбор типа кластера](../hdinsight-capacity-planning.md#choose-a-cluster-type).

* Если Ranger установлен в кластере, проверьте наличие слишком большого количества политик Ranger, которые необходимо вычислить для каждого запроса. Найдите дублирующиеся или ненужные политики.

* Проверьте значение **размера кучи HiveServer2** из Ambari. Выберите параметры   >  **конфигурации** Hive  >    >  **Оптимизация**. Убедитесь, что значение превышает 10 ГБ. Внесите необходимые изменения, чтобы оптимизировать производительность.

* Убедитесь, что запрос Hive настроен правильно. Дополнительные сведения см. [в статье Оптимизация запросов Apache Hive в Azure HDInsight](../hdinsight-hadoop-optimize-hive-query.md).

# <a name="collaborative-coding-with-git"></a>Совместное программирование с помощью Git

В этой статье описывается, как использовать Git в качестве платформы разработки для совместного использования кода для проектов обработки и анализа данных. В этой статье описано, как связать код в Azure Repos с рабочими элементами для [гибкой разработки](agile-development.md) в Azure Boards, как выполнять проверку кода, а также как создавать и объединять запросы на включение внесенных изменений.

## <a name="link-a-work-item-to-an-azure-repos-branch"></a><a name='Linkaworkitemwithagitbranch-1'></a>Связывание рабочего элемента с ветвью Azure Repos 

Azure DevOps предоставляет удобный способ подключения Azure Boards пользовательской истории или рабочего элемента задачи с помощью ветви репозитория Git Azure Repos. Вы можете связать пользовательскую историю или задачу непосредственно с кодом, связанным с ним. 

Чтобы подключить рабочий элемент к новой ветви, щелкните многоточие **действия** (**...**) рядом с рабочим элементом, а затем в контекстном меню прокрутите вниз до пункта **создать ветвь**.  

![1](./media/collaborative-coding-with-git/1-sprint-board-view.png)

В диалоговом окне **Создание ветви** укажите новое имя ветви и базовый Azure Repos репозиторий Git и Branch. Базовый репозиторий должен находиться в том же проекте Azure DevOps, что и рабочий элемент. Базовая ветвь может быть любой существующей ветвью. Выберите **создать ветвь**. 

![2](./media/collaborative-coding-with-git/2-create-a-branch.png)

Вы также можете создать новую ветвь, используя следующую команду git Bash в Windows или Linux:

```bash
git checkout -b <new branch name> <base branch name>

```
Если не указать \<base branch name> , новая ветвь будет основана на `main` . 

Чтобы переключиться на рабочую ветвь, выполните следующую команду: 

```bash
git checkout <working branch name>
```

После переключения в рабочую ветвь можно приступить к разработке кода или артефактов документации, чтобы завершить рабочий элемент. Выполняется `git checkout main` переключение обратно в `main` ветвь.

Рекомендуется создать ветвь Git для каждого рабочего элемента пользовательской истории. Затем для каждого рабочего элемента задачи можно создать ветвь на основе ветви пользовательской истории. Упорядочите ветви в иерархии, которая соответствует связи пользователя Story-Task, если несколько человек работают над разными пользовательскими историями для одного и того же проекта или для разных задач для одной и той же пользовательской истории. Конфликты можно сокращать, так как каждый участник команды работает в другой ветви или в другом коде или других артефактах при совместном использовании ветви. 

На следующей схеме показана Рекомендуемая стратегия ветвления для TDSP. Вам может не понадобиться столько ветвей, сколько показано здесь, особенно если в проекте работает только один или два человека или только один человек работает над всеми задачами пользовательской истории. Но отделение разработки от первичной ветви всегда является хорошей практикой и может помочь предотвратить прерывание ветви выпуска с помощью действий разработки. Полное описание модели ветвления Git см. [в разделе успешная модель ветвления Git](https://nvie.com/posts/a-successful-git-branching-model/).

![3](./media/collaborative-coding-with-git/3-git-branches.png)

Также вы можете связать рабочий элемент с существующей ветвью. На странице **сведений** рабочего элемента выберите **Добавить ссылку**. Затем выберите существующую ветвь, чтобы связать рабочий элемент, и нажмите кнопку **ОК**. 

![4](./media/collaborative-coding-with-git/4-link-to-an-existing-branch.png)

## <a name="work-on-the-branch-and-commit-changes"></a><a name='WorkonaBranchandCommittheChanges-2'></a>Работа с ветвью и фиксация изменений 

После внесения изменений в рабочий элемент, например при добавлении файла скрипта R в ветвь локального компьютера `script` , можно зафиксировать изменения из локальной ветви в вышестоящей рабочей ветви с помощью следующих команд git Bash:

```bash
git status
git add .
git commit -m "added an R script file"
git push origin script
```

![5](./media/collaborative-coding-with-git/5-sprint-push-to-branch.png)

## <a name="create-a-pull-request"></a><a name='CreateapullrequestonVSTS-3'></a>Создание запроса на включение внесенных изменений

После завершения одной или нескольких фиксаций и push-уведомлений, когда вы будете готовы объединить текущую рабочую ветвь в базовую ветвь, вы можете создать и отправить *запрос на вытягивание* в Azure Repos. 

На главной странице проекта Azure DevOps наведите указатель на **репозиториев**  >  **Pull (запросы на вытягивание** ) в левой области навигации. Затем выберите одну из **новых кнопок запроса на включение** внесенных изменений или ссылку **создать запрос на включение внесенных изменений** .

![6](./media/collaborative-coding-with-git/6-spring-create-pull-request.png)

При необходимости на экране **нового запроса на включение внесенных** изменений перейдите в репозиторий Git и ветвь, в которую необходимо объединить изменения. Добавьте или измените любые другие нужные сведения. В разделе **рецензенты** добавьте имена рецензентов, а затем выберите **создать**. 

![7](./media/collaborative-coding-with-git/7-spring-send-pull-request.png)

## <a name="review-and-merge"></a><a name='ReviewandMerge-4'></a>Проверка и слияние

После создания запроса на вытягивание рецензенты получают уведомление по электронной почте для просмотра запроса на вытягивание. Проверяющие проверяют, работают ли изменения, и по возможности проверяйте изменения с инициатором запроса. Рецензенты могут делать комментарии, запрашивать изменения, утверждать или отклонять запрос на вытягивание в зависимости от их оценки. 

![8](./media/collaborative-coding-with-git/8-add_comments.png)

После того как рецензенты утверждают изменения, вы или кто-то другой с разрешениями на слияние может объединить рабочую ветвь с базовой ветвью. Выберите **завершить**, а затем в диалоговом окне **полный запрос на включение внесенных изменений** выберите **завершить слияние** . После слияния рабочую ветвь можно удалить. 

![10](./media/collaborative-coding-with-git/10-spring-complete-pullrequest.png)

Убедитесь, что запрос помечен как **завершенный**. 

![11](./media/collaborative-coding-with-git/11-spring-merge-pullrequest.png)

Вернувшись к **репозиториев** в левой области навигации, можно увидеть, что вы перешли на главную ветвь с момента `script` удаления ветви.

![12](./media/collaborative-coding-with-git/12-spring-branch-deleted.png)

Вы также можете использовать следующие команды git bash для слияния `script` рабочей ветви с базовой ветвью и удаления рабочей ветви после слияния:

```bash
git checkout main
git merge script
git branch -d script
```

![13](./media/collaborative-coding-with-git/13-spring-branch-deleted-commandline.png)

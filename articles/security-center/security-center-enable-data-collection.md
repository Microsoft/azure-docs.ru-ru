# <a name="configure-auto-provisioning-for-agents-and-extensions-from-azure-security-center"></a>Настройка автоматической подготовки агентов и расширений из Центра безопасности Azure

Центр безопасности Azure собирает данные из ресурсов с помощью соответствующего агента или расширения для этого ресурса и включенного типа коллекции данных. Выполните указанные ниже процедуры, чтобы убедиться, что ресурсы имеют необходимые агенты и расширения, используемые Центром безопасности.

## <a name="prerequisites"></a>Предварительные требования
Для начала работы с Центром безопасности необходима подписка Microsoft Azure. Если у вас нет ее, вы можете [зарегистрироваться для получения бесплатной учетной записи](https://azure.microsoft.com/pricing/free-trial/).

## <a name="how-does-security-center-collect-data"></a>Как Центр безопасности собирает данные?

Центр безопасности собирает данные c ваших виртуальных машин Azure, масштабируемых наборов виртуальных машин, контейнеров IaaS и компьютеров, не относящихся к Azure (в том числе локальных), чтобы отслеживать уязвимости и угрозы безопасности. 

Сбор данных необходим, чтобы обнаруживать недостающие обновления и неправильно настроенные параметры безопасности ОС, узнавать, включена ли защита конечных точек, а также определять работоспособность и обнаруживать угрозы. Сбор данных необходим только для ресурсов вычислительной среды (например, виртуальных машин, масштабируемых наборов виртуальных машин, контейнеров IaaS и компьютеров, не относящихся к Azure). 

Вы можете пользоваться центром безопасности Azure даже без подготовки агентов. Однако защита будет ограничена, а перечисленные выше возможности не будут поддерживаться.  

Для сбора данных используются указанные ниже возможности.

- **Агент Log Analytics**, который считывает различные конфигурации, связанные с безопасностью, и журналы событий с компьютера, а также копирует данные в рабочую область для анализа. К примерам таких данных относятся тип и версия операционной системы, журналы операционной системы (журналы событий Windows), выполняющиеся процессы, имя компьютера, IP-адреса и имя вошедшего пользователя.
- **Модули безопасности**, такие как [надстройка Политики Azure для Kubernetes](../governance/policy/concepts/policy-for-kubernetes.md), которые также могут передавать данные по специализированным типам ресурсов в Центр безопасности.

> [!TIP]
> По мере увеличения объема Центра безопасности увеличивается число типов ресурсов, которые можно отслеживать. Число расширений также увеличивается. Средства автоматической подготовки расширены. Теперь они поддерживают дополнительные типы ресурсов, используя возможности Политики Azure.

## <a name="why-use-auto-provisioning"></a>Преимущества автоматической подготовки
Любые агенты и расширения, описанные на этой странице, *можно* установить вручную (см. раздел о [ручной установке агента Log Analytics](#manual-agent)). Однако **автоматическая подготовка** сокращает расходы на управление, устанавливая все необходимые агенты и расширения на существующих и новых компьютерах, чтобы обеспечить своевременный уровень безопасности для всех поддерживаемых ресурсов. 

Рекомендуется включить автоматическую подготовку. По умолчанию она отключена.

## <a name="how-does-auto-provisioning-work"></a>Принцип работы автоматической подготовки
Параметры автоматической подготовки Центра безопасности содержат переключатель для каждого поддерживаемого типа расширения. При включении автоматической подготовки расширения вы назначаете соответствующую политику **Deploy if not exists** (Развернуть, если не существует), чтобы убедиться, что расширение подготовлено на всех существующих и будущих ресурсах этого типа.

> [!TIP]
> Узнайте больше об эффекте Политики Azure, а также политики Deploy if not exists (Развернуть, если не существует), из [этой статьи](../governance/policy/concepts/effects.md).


## <a name="enable-auto-provisioning-of-the-log-analytics-agent-and-extensions"></a>Включение автоматической подготовки агента и расширений Log Analytics <a name="auto-provision-mma"></a>

Если автоматическая подготовка включена для агента Log Analytics, то Центр безопасности развертывает агент на всех поддерживаемых и на созданных виртуальных машинах Azure. 1. Выберите соответствующую подписку.

## <a name="windows-security-event-options-for-the-log-analytics-agent"></a>Параметры событий безопасности Windows для Log Analytics Agent <a name="data-collection-tier"></a> 

Выбор уровня сбора данных в Центре безопасности Azure повлияет только на *хранение* событий безопасности в рабочей области Log Analytics. Log Analytics Agent по-прежнему будет собирать и анализировать события безопасности, необходимые для защиты от угроз в Центре безопасности, независимо от того, какой уровень событий безопасности вы хотите сохранить в рабочей области. Выбор хранения событий безопасности позволит проводить расследование, поиск и аудит этих событий в вашей рабочей области.

### <a name="requirements"></a>Требования 
Для хранения данных событий безопасности Windows требуется Azure Defender. Дополнительные сведения об Azure Defender см. в [этой статье](azure-defender.md).

Сохранение данных в Log Analytics может привести к дополнительным расходам на хранение данных. Дополнительные сведения см. на [странице с расценками](https://azure.microsoft.com/pricing/details/security-center/).

### <a name="information-for-azure-sentinel-users"></a>Сведения для пользователей Azure Sentinel 
Пользователям Azure Sentinel нужно обратить внимание, что сбор данных о событиях безопасности в контексте одной рабочей области следует настраивать только в одном месте: либо в Центре безопасности Azure, либо в Azure Sentinel. Если вы планируете добавить Azure Sentinel в рабочую область, которая уже получает оповещения из Центра безопасности Azure и настроена на сбор данных о событиях безопасности, у вас есть два варианта:
- Не меняйте способ сбора данных о событиях безопасности в Центре безопасности Azure. Так вы сможете запрашивать и анализировать эти события как в Azure Sentinel, так и в Azure Defender. Однако вы не сможете отслеживать состояние подключения соединителя или изменять его конфигурацию в Azure Sentinel. Если это важно для вас, рассмотрите второй вариант.
- Отключите сбор данных о событиях безопасности в Центре безопасности Azure (задав для параметра **События безопасности Windows** значение **Нет** в конфигурации агента Log Analytics). Затем добавьте соединитель событий безопасности в Azure Sentinel. Как и в первом варианте, вы сможете запрашивать и анализировать события как в Azure Sentinel, так и в Azure Defender или Центре безопасности Azure, а также отслеживать состояние подключения соединителя или изменять его конфигурацию в Azure Sentinel.

### <a name="what-event-types-are-stored-for-common-and-minimal"></a>Какие типы событий сохраняются для наборов "Общий" и "Минимальный"?
Эти наборы предназначены для стандартных сценариев. Выберите набор, который соответствует вашим требованиям, прежде чем внедрить его.

Для определения событий, которые будут принадлежать к вариантам **Общий** и **Минимальный**, мы ознакомились с отзывами клиентов и отраслевыми стандартами, чтобы изучить нефильтрованную частоту каждого события и их использования. В процессе мы использовали приведенные ниже рекомендации.

- **Минимальный.** Убедитесь, что этот набор содержит только события, которые могут указывать на брешь в системе безопасности и важные события небольших объемов. Например, этот набор содержит относительно большой объем сведений об успешных и неудачных попытках входа пользователя (идентификаторы события 4624 и 4625), однако не содержит сведений о выходе, что очень важно для аудита и не важно для обнаружения. Большую часть данных этого набора занимают события входа и создания процессов (идентификатор событий 4688).
- **Общий.** В этом наборе предоставляется полный журнал аудита пользователя. Например, этот набор содержит данные о входе и выходе пользователя (идентификатор события 4634). Мы добавили действия аудита, такие как изменение группы безопасности, операции основного контроллера домена Kerberos и другие события, рекомендованные отраслевыми организациями.

События небольших объемов были включены в набор "Общий". Основной причиной для его выбора среди всех событий является возможность уменьшить объем и оставить определенные события нефильтрованными.

### <a name="setting-the-security-event-option-at-the-workspace-level"></a>Настройка параметра события безопасности на уровне рабочей области

Вы можете определить уровень данных событий безопасности, которые будут храниться на уровне рабочей области.

1. В меню Центра безопасности на портале Azure выберите **Цены и параметры**.
1. Выберите соответствующую рабочую область. Единственными событиями сбора данных для рабочей области являются события безопасности Windows, описанные на этой странице.

    :::image type="content" source="media/security-center-enable-data-collection/event-collection-workspace.png" alt-text="Настройка данных событий безопасности для хранения в рабочей области":::

1. Выберите объем необработанных данных события для хранения, а затем — **Сохранить**.

## <a name="manual-agent-provisioning"></a>Подготовка агента вручную <a name="manual-agent"></a>
 
Чтобы вручную установить агент Log Analytics, выполните указанные ниже действия.

1. Отключите автоматическую подготовку.

1. При необходимости создайте рабочую область.

1. Включите Azure Defender в рабочей области, в которой устанавливаете агент Log Analytics:

    1. В главном меню Центра безопасности выберите **Цены и параметры**.

    1. Укажите рабочую область, в которой устанавливаете агент. Убедитесь, что рабочая область находится в той же подписке, которая используется в Центре безопасности, а у вас есть разрешения на чтение и запись в этой рабочей области.

    1. Выберите **Включить Azure Defender** и **Сохранить**.

       >[!NOTE]
       >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 

1. Чтобы развернуть агенты на новых виртуальных машинах с помощью шаблона Resource Manager, установите агент Log Analytics:

   - см. сведения об [установке агента Log Analytics для Windows](../virtual-machines/extensions/oms-windows.md);
   - см. сведения об [установке агента Log Analytics для Linux](../virtual-machines/extensions/oms-linux.md).

1. Чтобы развернуть агенты на существующих виртуальных машинах, следуйте инструкциям из статьи [Сбор данных о виртуальных машинах Azure](../azure-monitor/vm/quick-collect-azurevm.md) (раздел о **сборе данных о событиях и производительности** является необязательным).

1. Чтобы развернуть агенты с помощью PowerShell, следуйте инструкциям из документации по виртуальным машинам:

    - [Для компьютеров Windows](../virtual-machines/extensions/oms-windows.md?toc=%2fazure%2fazure-monitor%2ftoc.json#powershell-deployment)
    - [Для компьютеров Linux](../virtual-machines/extensions/oms-linux.md?toc=%2fazure%2fazure-monitor%2ftoc.json#azure-cli-deployment)

> [!TIP]
> Инструкции о том, как подключиться к центру безопасности с помощью PowerShell, см. в разделе [Automate onboarding of Azure Security Center using PowerShell](security-center-powershell-onboarding.md) (Автоматизация подключения центра безопасности Azure с помощью PowerShell).


## <a name="automatic-provisioning-in-cases-of-a-pre-existing-agent-installation"></a>Автоматическая подготовка уже существующей установки агента <a name="preexisting"></a> 

В описанных ниже вариантах использования указано, как автоматическая подготовка работает в тех случаях, когда агент или расширение уже установлены. 

- **Агент Log Analytics непосредственно установлен на компьютере, но не в качестве расширения (прямой агент)** . Если агент Log Analytics установлен непосредственно на виртуальной машине (не в качестве расширения Azure), Центр безопасности установит расширение агента Log Analytics и сможет обновить агент Log Analytics до последней версии.
Установленный агент продолжит передавать данные в уже настроенные рабочие области и в дополнение будет передавать их в рабочую область, настроенную в Центре безопасности (на компьютерах Windows поддерживается одновременная работа с несколькими поставщиками услуг).
Если настроенная рабочая область является пользовательской (не рабочей областью Центра безопасности по умолчанию), вам потребуется установить в ней решение Security или SecurityCenterFree для Центра безопасности, чтобы начать обработку событий с виртуальных машин и компьютеров, передающих данные в эту рабочую область.

    Для компьютеров Linux одновременная работа с несколькими поставщиками услуг еще не поддерживается, поэтому при обнаружении существующей установки агента автоматическая подготовка не выполняется, и конфигурация компьютера не будет изменена.

    Для существующих компьютеров в подписках, подключенных к Центру безопасности до 17 марта 2019 года, при обнаружении существующего агента расширение агента Log Analytics не будет установлено и конфигурация компьютера не будет изменена. Рекомендации по устранению проблем установки агента на таких компьютерах см. в разделе "Устранение проблем работоспособности агента мониторинга на ваших компьютерах".
  
- **Агент System Center Operations Manager уже установлен на компьютере**. Центр безопасности установит расширение агента Log Analytics параллельно с существующим агентом Operations Manager. Существующий агент Operations Manager продолжит отправлять отчеты на сервер Operations Manager, как обычно. Агент Operations Manager и агент Log Analytics используют общие библиотеки времени выполнения, которые во время этого процесса будут обновлены до последней версии. Если установлен агент Operations Manager версии 2012, **не** включайте автоматическую подготовку.

- **Расширение виртуальной машины уже установлено**:
    - Если агент мониторинга установлен в качестве расширения, конфигурация расширения позволяет создавать отчеты только для одной рабочей области. Центр безопасности не переопределяет установленные подключения к рабочим областям пользователя. Центр безопасности будет хранить данные безопасности с виртуальной машины в уже подключенной рабочей области при условии, что в ней установлено решение Security или SecurityCenterFree. В рамках этого процесса Центр безопасности может обновлять расширение до последней версии.
    - Чтобы узнать, в какую рабочую область отправляет данные имеющееся расширение, выполните тест для [проверки соединения с центром безопасности Azure](/archive/blogs/yuridiogenes/validating-connectivity-with-azure-security-center). Вы также можете открыть рабочие области Log Analytics, выбрать рабочую область и виртуальную машину, а затем проверить подключение агента Log Analytics.
    - Если в вашей среде на клиентских рабочих станциях установлен агент Log Analytics, отправляющий отчеты в имеющуюся рабочую область Log Analytics, просмотрите список [операционных систем, поддерживаемых Центром безопасности Azure](security-center-os-coverage.md), чтобы убедиться, что ваша операционная система поддерживается. Дополнительные сведения см. в статье [Существующие клиенты Log Analytics](./faq-azure-monitor-logs.md).
 

## <a name="disable-auto-provisioning"></a>Отключение автоматической подготовки <a name="offprovisioning"></a>

При отключении автоматической подготовки агенты не будут подготавливаться на новых виртуальных машинах.

Чтобы отключить автоматическую подготовку агента, выполните указанные ниже действия.

1. В меню центра безопасности на портале выберите **Цены и параметры**.
1. Выберите соответствующую подписку.
1. Выберите **Автоматическая подготовка**.
1. Установите для состояния соответствующего агента значение **Выкл.**

    :::image type="content" source="./media/security-center-enable-data-collection/agent-toggles.png" alt-text="Переключение состояния, чтобы отключить автоматическую подготовку для каждого типа агента":::

1. Щелкните **Сохранить**. Если автоматическая подготовка отключена, раздел конфигурации рабочей области по умолчанию не отображается:

    :::image type="content" source="./media/security-center-enable-data-collection/empty-configuration-column.png" alt-text="Если автоматическая подготовка отключена, ячейка конфигурации пуста":::


> [!NOTE]
>  При отключении автоматической подготовки агент Log Analytics не будет удален с виртуальных машин Azure, на которых он был подготовлен. Сведения о том, как удалить расширение OMS, см. в разделе [Как удалить расширения OMS, установленные центром безопасности](faq-data-collection-agents.md#remove-oms).
>

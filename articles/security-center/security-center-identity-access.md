# <a name="manage-multi-factor-authentication-mfa-enforcement-on-your-subscriptions"></a>Управление принудительной проверкой многофакторной идентификации (MFA) в подписках

Если вы используете только пароли для проверки подлинности пользователей, вы оставите вектор атаки открытым. Пользователи часто используют ненадежные пароли или могут повторно использоваться для нескольких служб. Включение [Многофакторной идентификации](https://www.microsoft.com/security/business/identity/mfa) (MFA) повышает безопасность учетных записей, и пользователи по-прежнему смогут проходить аутентификацию практически в любом приложении благодаря единому входу.

Включить MFA для пользователей Azure Active Directory (AD) в зависимости от лицензий, которыми владеет ваша организация, можно несколькими способами. На этой странице приведены подробные сведения о каждом из в контексте центра безопасности Azure.


## <a name="mfa-and-security-center"></a>MFA и центр безопасности 

Центр безопасности помещает большое значение в MFA. Управление безопасностью, влияющее на оценку безопасности, **включает MFA**. 

Рекомендации по включению элемента управления MFA обеспечивают соответствие рекомендациям для пользователей ваших подписок:

- MFA должна быть включена для учетных записей с разрешениями владельца в вашей подписке.
- MFA должна быть включена для учетных записей с разрешениями на запись в вашей подписке.

Существует три способа включить MFA и обеспечить соответствие двум рекомендациям в центре безопасности: параметры безопасности по умолчанию, назначение на пользователя, политика условного доступа (CA). Каждый из этих вариантов описан ниже.

### <a name="free-option---security-defaults"></a>Бесплатный вариант — параметры безопасности по умолчанию
Если вы используете бесплатный выпуск Azure AD, используйте [Параметры безопасности по умолчанию](../active-directory/fundamentals/concept-fundamentals-security-defaults.md) , чтобы включить многофакторную проверку подлинности в клиенте.

### <a name="mfa-for-microsoft-365-business-e3-or-e5-customers"></a>MFA для клиентов Microsoft 365 бизнес, E3 или
Клиенты с Microsoft 365 могут использовать **назначение "на пользователя**". В этом случае Azure AD MFA включена или отключена для всех пользователей для всех событий входа. Невозможно включить многофакторную проверку подлинности для подмножества пользователей или в определенных сценариях, а управление осуществляется через портал Office 365.

### <a name="mfa-for-azure-ad-premium-customers"></a>MFA для клиентов Azure AD Premium
Для повышения удобства работы пользователей выполните обновление до Azure AD Premium P1 или P2 для параметров **политики условного доступа (CA)** . Чтобы настроить политику ЦС, необходимы [разрешения клиента Azure Active Directory (AD)](../active-directory/roles/permissions-reference.md).

Политика ЦС должна:
- применение MFA
- включить идентификатор приложения управления Microsoft Azure (797f4846-ba00-4fd7-ba43-dac1f8f63013) или все приложения
- Идентификатор приложения управления Microsoft Azure не исключен

Клиенты **Azure AD Premium P1** могут использовать ЦС Azure AD для запроса многофакторной проверки подлинности во время определенных сценариев или событий в соответствии с бизнес-требованиями. Другие лицензии, включающие эту функцию: Enterprise Mobility + Security E3, Microsoft 365 F1 и Microsoft 365 E3.

**Azure AD Premium P2** предоставляет самые надежные функции обеспечения безопасности и улучшенное взаимодействие с пользователем. Эта лицензия добавляет [Условный доступ на основе рисков](../active-directory/conditional-access/howto-conditional-access-policy-risk.md) к функциям Azure AD Premium P1. ЦС на основе рисков адаптируется к шаблонам пользователей и свертывает запросы многофакторной проверки подлинности. Другие лицензии, включающие эту функцию: Enterprise Mobility + Security или "е" или "Microsoft 365".

Дополнительные сведения см. в [документации по условному доступу Azure](../active-directory/conditional-access/overview.md).

## <a name="identify-accounts-without-multi-factor-authentication-mfa-enabled"></a>Идентификация учетных записей без поддержки многофакторной проверки подлинности (MFA)

Список учетных записей пользователей без поддержки MFA можно просмотреть на странице с подробными сведениями о рекомендациях центра безопасности или с помощью графа ресурсов Azure.

### <a name="view-the-accounts-without-mfa-enabled-in-the-azure-portal"></a>Просмотр учетных записей без поддержки MFA в портал Azure
На странице сведения об рекомендации выберите подписку в списке **неработоспособные ресурсы** или выберите **действие выполнить** , и список будет отображаться.

### <a name="view-the-accounts-without-mfa-enabled-using-azure-resource-graph"></a>Просмотр учетных записей без поддержки MFA с помощью графа ресурсов Azure
Чтобы узнать, какие учетные записи не включены в MFA, используйте следующий запрос графа ресурсов Azure. Запрос возвращает все неработоспособные ресурсы — учетные записи — из рекомендации "MFA должно быть включено для учетных записей с разрешениями владельца для вашей подписки". 

1. Откройте **Обозреватель графов ресурсов Azure**.

    :::image type="content" source="./media/security-center-identity-access/opening-resource-graph-explorer.png" alt-text="Страница рекомендаций по запуску обозревателя графа ресурсов Azure * *" :::

1. Введите следующий запрос и выберите **выполнить запрос**.

    ```kusto
    securityresources
     | where type == "microsoft.security/assessments"
     | where properties.displayName == "MFA should be enabled on accounts with owner permissions on your subscription"
     | where properties.status.code == "Unhealthy"
    ```

1. `additionalData`Свойство показывает список идентификаторов объектов учетных записей, для которых не применяется аутентификация mfa. 

    > [!NOTE]
    > Учетные записи отображаются в виде идентификаторов объектов, а не имен учетных записей для защиты конфиденциальности владельцев учетных записей.

> [!TIP]
> Кроме того, вы можете использовать оценку метода REST API для центра безопасности [— получить](/rest/api/securitycenter/assessments/get).

### <a name="were-already-using-ca-policy-to-enforce-mfa-why-do-we-still-get-the-security-center-recommendations"></a>Мы уже используем политику ЦС для принудительного применения MFA. Почему мы по-прежнему получаем рекомендации центра безопасности?
Чтобы выяснить, почему рекомендации по-прежнему создаются, проверьте следующие параметры конфигурации в политике ЦС MFA:

- Вы включили учетные записи в раздел " **Пользователи** " в политике "ЦС MFA" (или одну из групп в разделе " **группы** ").
- Идентификатор приложения управления Azure (797f4846-ba00-4fd7-ba43-dac1f8f63013) или все приложения включены в раздел **приложений** политики центра сертификации mfa.
- Идентификатор приложения управления Azure не исключен в разделе **приложений** политики ЦС mfa.

### <a name="were-using-a-third-party-mfa-tool-to-enforce-mfa-why-do-we-still-get-the-security-center-recommendations"></a>Мы используем стороннее средство MFA для применения MFA. Почему мы по-прежнему получаем рекомендации центра безопасности?
Рекомендации по MFA в центре безопасности не поддерживают сторонние средства MFA (например, DUO).

Если рекомендации для вашей организации не используются, попробуйте пометить их как "устраненные", как описано в разделе [исключение ресурсов и рекомендаций из оценки безопасности](exempt-resource.md). Также можно [Отключить рекомендацию](tutorial-security-policy.md#disable-security-policies-and-disable-recommendations).

### <a name="why-does-security-center-show-user-accounts-without-permissions-on-the-subscription-as-requiring-mfa"></a>Почему центр безопасности отображает учетные записи пользователей без разрешений на подписку как "требует MFA"?
Рекомендации по MFA для центра безопасности относятся к ролям [Azure RBAC](../role-based-access-control/role-definitions-list.md) и роли [администраторов классической подписки Azure](../role-based-access-control/classic-administrators.md) . Убедитесь, что ни одна из учетных записей не имеет таких ролей.

### <a name="were-enforcing-mfa-with-pim-why-are-pim-accounts-shown-as-noncompliant"></a>Мы принудительно используем MFA с PIM. Почему учетные записи PIM отображаются как несоответствующие требованиям?
В настоящее время рекомендации по MFA для центра безопасности не поддерживают учетные записи PIM. Эти учетные записи можно добавить в политику условного доступа в раздел пользователей или группы.

### <a name="can-i-exempt-or-dismiss-some-of-the-accounts"></a>Можно ли исключить или отклонить некоторые учетные записи?
В настоящее время не поддерживается возможность исключения некоторых учетных записей, которые не используют MFA.  

### <a name="are-there-any-limitations-to-security-centers-identity-and-access-protections"></a>Существуют ли ограничения для защиты идентификации и доступа в центре безопасности?
Существуют некоторые ограничения для защиты идентификации и доступа в центре безопасности.

- Рекомендации по удостоверениям недоступны для подписок с более чем 600 учетными записями. В таких случаях эти рекомендации будут перечислены в разделе "недоступные оценки".
- Рекомендации по идентификации недоступны для агентов администрирования партнера поставщика облачных решений (CSP).
- Рекомендации по идентификации не определяют учетные записи, управляемые с помощью системы управления привилегированными пользователями (PIM). Если вы используете средство PIM, вы можете увидеть неточные результаты в элементе управления **доступом и разрешениями** .

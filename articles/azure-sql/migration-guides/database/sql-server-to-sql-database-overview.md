# <a name="migration-overview-sql-server-to-sql-database"></a>Общие сведения о миграции: SQL Server в базу данных SQL
[!INCLUDE[appliesto--sqldb](../../includes/appliesto-sqldb.md)]

Узнайте о различных вариантах миграции и особенностях переноса SQL Server в базу данных SQL Azure. 

Вы можете выполнить миграцию SQL Server, выполняемого локально или в: 

- SQL Server на виртуальных машинах  
- Amazon Web Services (AWS) EC2 
- Служба реляционной базы данных Amazon (AWS RDS) 
- Подсистема вычислений (Google Cloud Platform-обеспечить)  
- Облачный SQL для SQL Server (Google Cloud Platform – обеспечить) 

Другие сценарии см. в разделе [руководств по миграции баз данных](https://datamigration.microsoft.com/). 

## <a name="overview"></a>Обзор

[База данных SQL Azure](../../database/sql-database-paas-overview.md) является рекомендуемым вариантом для рабочих нагрузок SQL Server, которым требуется полностью управляемая платформа как услуга (PaaS). База данных SQL обрабатывает большинство функций управления базами данных, а также высокую доступность, интеллектуальную обработку запросов, масштабируемость и возможности производительности, встроенные в соответствие множеству различных типов приложений. 

База данных SQL обеспечивает гибкость с несколькими [моделями развертывания](../../database/sql-database-paas-overview.md#deployment-models) и [уровнями служб](../../database/service-tiers-vcore.md#service-tiers) , предназначенными для различных типов приложений или рабочих нагрузок.

Одним из ключевых преимуществ перехода на базу данных SQL является то, что вы можете модернизировать приложение, используя возможности PaaS и исключая все зависимости от технических компонентов, которые находятся на уровне экземпляра, например в заданиях агента SQL.

Вы также можете сэкономить затраты, переполнив SQL Server локальные лицензии в базу данных SQL Azure, используя [преимущество гибридного использования Azure](https://azure.microsoft.com/pricing/hybrid-benefit/) для SQL Server следует выбрать [модель приобретения на основе виртуальное ядро](../../database/service-tiers-vcore.md).

Это пошаговое руководством предназначено для уточнения вариантов миграции и рекомендаций по подготовке к переносу баз данных SQL Server в базу данных SQL Azure.  

## <a name="considerations"></a>Рекомендации 

Ключевые факторы, которые следует учитывать при оценке параметров миграции, зависят от следующих факторов. 

- Количество серверов и баз данных
- Размер баз данных
- Приемлемый простой в работе в процессе миграции 

Параметры миграции, перечисленные в этом пошаговом окне, учитываются при работе с этими факторами. Для миграции логических данных в базу данных SQL Azure время миграции может зависеть как от количества объектов в базе данных, так и от размера базы данных. 

Для разных рабочих нагрузок и настроек пользователей доступны различные средства. Некоторые средства можно использовать для быстрой миграции отдельной базы данных с помощью средства на основе пользовательского интерфейса, а другие средства могут выполнять миграцию нескольких баз данных, которые можно автоматизировать для выполнения миграции в масштабе. 

## <a name="choose-appropriate-target"></a>Выберите подходящий целевой объект

Рассмотрите общие рекомендации, которые помогут вам выбрать правильную модель развертывания и уровень обслуживания базы данных SQL Azure. Вы можете выбрать ресурсы для вычислений и хранения во время развертывания, а затем изменить их позже, используя  [портал Azure](../../database/scale-resources.md)  без простоя приложения.


**Модели развертывания**. Изучите рабочую нагрузку приложения и шаблон использования, чтобы выбрать одну базу данных или эластичный пул. 

- [Одна база данных](../../database/single-database-overview.md) представляет полностью управляемую базу данных, подходящую для большинства современных облачных приложений и микрослужб.
- [Эластичный пул](../../database/elastic-pool-overview.md) — это набор отдельных баз данных с общим набором ресурсов, таких как ЦП или память, который подходит для объединения баз данных в пуле с прогнозируемыми шаблонами использования, которые могут эффективно использовать один и тот же набор ресурсов.

**Модели приобретения**: Выберите виртуальное ядро, DTU или модель бессерверного приобретения. 

- [Модель Виртуальное ядро](../../database/service-tiers-vcore.md) позволяет выбрать количество виртуальных ядер для базы данных SQL Azure, что делает ее самым простым выбором при переводе из локальной SQL Server. Это единственный вариант, который поддерживает сохранение по стоимости лицензии с [преимущество гибридного использования Azure](https://azure.microsoft.com/pricing/hybrid-benefit/). 
- [Модель DTU](../../database/service-tiers-dtu.md) абстрагирует базовые ресурсы вычислений, памяти и операций ввода-вывода, чтобы обеспечить смешивание DTU. 
- [Бессерверная модель](../../database/serverless-tier-overview.md) предназначена для рабочих нагрузок, которым требуется автоматическое масштабирование по запросу с расчетными ресурсами за секунду использования. Уровень вычислений "бессерверный" автоматически приостанавливает базы данных в течение неактивных периодов (в которых оплачивается только хранилище) и автоматически возобновляет базы данных при возвращении действия. 

**Уровни служб**. Выберите один из трех уровней служб, предназначенных для различных типов приложений.

- [Общего назначения или уровень служб "Стандартный](../../database/service-tier-general-purpose.md) " предлагает сбалансированный параметр, ориентированный на бюджет, с помощью вычислений и хранилища, пригодных для создания приложений среднего уровня с избыточностью на уровне хранилища для восстановления после сбоев. Предназначен для большинства рабочих нагрузок базы данных. 
- [Уровень служб критически важный для бизнеса/Premium](../../database/service-tier-business-critical.md) предназначен для высокоуровневого приложения, требующих высокой скорости транзакций, операций ввода-вывода с низкой задержкой и высокого уровня устойчивости с вторичными репликами, доступными для отработки отказа и для разгрузки рабочих нагрузок чтения.
- [Уровень службы "горизонтальное масштабирование](../../database/service-tier-hyperscale.md) " предназначен для баз данных с растущими объемами данных и должен автоматически масштабироваться до 100 ТБ в размере базы данных. Предназначен для очень больших баз данных. 

> [!IMPORTANT]
> [Скорость журнала транзакций регулируется](../../database/resource-limits-logical-server.md#transaction-log-rate-governance) в базе данных SQL Azure для ограничения высокой скорости приема. Таким образом, во время миграции может потребоваться масштабировать ресурсы целевой базы данных (виртуальных ядер/DTU), чтобы снизить нагрузку на ЦП или пропускную способность. Выберите целевую базу данных соответствующего размера, но при необходимости запланируйте масштабирование ресурсов до миграции. 


### <a name="sql-server-on-azure-vm-alternative"></a>Альтернативная SQL Server виртуальной машины Azure

У вашего бизнеса могут быть требования, которые делают [SQL Server на виртуальных машинах Azure](../../virtual-machines/windows/sql-server-on-azure-vm-iaas-what-is-overview.md) более подходящим целевым объектом, чем база данных SQL Azure. 

Если следующее применимо к вашему бизнесу, рассмотрите возможность перехода на SQL Server на виртуальной машине Azure: 

- Если требуется прямой доступ к операционной системе или файловой системе, например для установки сторонних или пользовательских агентов на одной виртуальной машине с SQL Server. 
- При наличии достаточной зависимости от компонентов, которые по-прежнему не поддерживаются, таких как FileStream/FileTable, Polybase и транзакции между экземплярами. 
- Если вам абсолютно нужно остаться в определенной версии SQL Server (например, 2012). 
- Если требования к вычислительной мощности значительно ниже, чем предложения управляемого экземпляра (один Виртуальное ядро, например), и консолидация базы данных не является приемлемым вариантом.

## <a name="feature-interoperability"></a>Взаимодействие компонентов 

При переносе рабочих нагрузок, зависящих от других функций SQL Server, необходимо учитывать дополнительные моменты.

#### <a name="sql-server-integration-services"></a>SQL Server Integration Services
Перенос пакетов SQL Server Integration Services (SSIS) в Azure путем повторного развертывания пакетов в среде выполнения Azure-SSIS в [фабрике данных Azure](../../../data-factory/introduction.md). Фабрика данных Azure [поддерживает миграцию пакетов служб SSIS](../../../data-factory/scenario-ssis-migration-overview.md#azure-sql-database-as-database-workload-destination) , предоставляя среду выполнения, созданную для выполнения пакетов служб SSIS в Azure. Кроме того, можно также переписать логику ETL служб SSIS в ADF-файле с помощью [потоков](../../../data-factory/concepts-data-flow-overview.md)данных.


#### <a name="sql-server-reporting-services"></a>службы SQL Server Reporting Services
Миграция отчетов SQL Server Reporting Services (SSRS) в отчеты с разбивкой на страницы в Power BI. Используйте [средство миграции языка определения отчетов](https://github.com/microsoft/RdlMigration) для подготовки и переноса отчетов. Это средство было разработано корпорацией Майкрософт, чтобы помочь клиентам перенести RDL-отчеты с серверов SSRS на платформу Power BI. Он доступен на сайте GitHub, и он документирует полное пошаговое руководство по сценарию миграции. 

#### <a name="high-availability"></a>Высокий уровень доступности
Ручная настройка SQL Server высокого уровня доступности, таких как Always On экземпляров отказоустойчивого кластера и группы доступности Always On, устаревает в целевой базе данных SQL Azure, так как архитектура с высоким уровнем доступности уже встроена в [общего назначения (модель доступности уровня "Стандартный")](../../database/high-availability-sla.md#basic-standard-and-general-purpose-service-tier-locally-redundant-availability) и [критически важный для бизнеса (модель доступности уровня "Премиум")](../../database/high-availability-sla.md#premium-and-business-critical-service-tier-locally-redundant-availability) . Уровень служб критически важный для бизнеса/Premium также обеспечивает горизонтальное масштабирование чтения, позволяющее подключаться к одному из вторичных узлов для целей только для чтения. 

Помимо архитектуры высокого уровня доступности, включенной в базу данных SQL, существует также функция [групп автоматической отработки отказа](../../database/auto-failover-group-overview.md) , которая позволяет управлять репликацией и отработкой отказа баз данных в управляемом экземпляре в другой регион. 

#### <a name="sql-agent-jobs"></a>Задания агента SQL Server
Задания агента SQL напрямую не поддерживаются в базе данных SQL Azure и должны быть развернуты в [заданиях эластичной базы данных (Предварительная версия)](../../database/job-automation-overview.md).

#### <a name="logins-and-groups"></a>Имена входа и группы
Перемещение имен входа SQL из исходного SQL Server в базу данных SQL Azure с помощью Database Migration Service (DMS) в автономном режиме.  Используйте колонку " **выбранные имена входа** " в **мастере миграции** , чтобы перенести имена входа в целевую базу данных SQL. 

Пользователи и группы Windows также можно перенести с помощью DMS, включив соответствующий выключатель на странице конфигурации DMS. 

Кроме того, можно использовать [служебную программу PowerShell](https://github.com/microsoft/DataMigrationTeam/tree/master/IP%20and%20Scripts/MoveLogins) , разработанную специально для архитекторов Microsoft Data Migration Architect. Программа использует PowerShell для создания скрипта Transact-SQL (T-SQL) для повторного создания имен входа и выбора пользователей базы данных из источника в целевой объект. Средство автоматически сопоставляет учетные записи Windows AD с учетными записями Azure AD и может выполнить поиск имени участника-пользователя для каждого входа в исходную Active Directory. Средство создает скрипты для пользовательских ролей сервера и базы данных, а также для членства в роли, роли базы данных и разрешений пользователей. Автономные базы данных пока не поддерживаются, и скрипты имеют только подмножество возможных SQL Server разрешений. 


#### <a name="system-databases"></a>Системные базы данных
Для базы данных SQL Azure единственными применимыми системными базами данных являются [master](/sql/relational-databases/databases/master-database) и tempdb. Дополнительные сведения см. [в разделе tempdb в базе данных SQL Azure](/sql/relational-databases/databases/tempdb-database#tempdb-database-in-sql-database).

## <a name="leverage-advanced-features"></a>Использование дополнительных функций 

Не забудьте воспользоваться преимуществами расширенных облачных функций, предлагаемых базой данных SQL. Например, больше не нужно беспокоиться об управлении резервными копиями, так как служба делает это самостоятельно. Можно выполнить восстановление в любой [момент времени в течение срока хранения](../../database/recovery-using-backups.md#point-in-time-restore). 

Для усиления безопасности рекомендуется использовать [проверку Подлинности Azure Active Directory](../../database/authentication-aad-overview.md), [Аудит](../../database/auditing-overview.md), [обнаружение угроз](../../database/azure-defender-for-sql.md), [безопасность на уровне строк](/sql/relational-databases/security/row-level-security)и [Динамическое маскирование данных](/sql/relational-databases/security/dynamic-data-masking).

В дополнение к расширенным функциям управления и безопасности база данных SQL предоставляет набор дополнительных средств, которые могут помочь [в мониторинге и настройке рабочей нагрузки](../../database/monitor-tune-overview.md). [Аналитика SQL Azure (Предварительная версия)](../../../azure-monitor/insights/azure-sql.md) — это расширенное облачное решение для мониторинга производительности всех баз данных в базе данных SQL Azure в масштабе и нескольких подписок в одном представлении. Аналитика SQL Azure собирает и визуализирует ключевые метрики производительности со встроенной аналитикой для устранения неполадок с производительностью.

[Автоматическая настройка](/sql/relational-databases/automatic-tuning/automatic-tuning#automatic-plan-correction)   непрерывно отслеживает производительность статистики плана выполнения SQL и автоматически устраняет выявленные проблемы с производительностью.

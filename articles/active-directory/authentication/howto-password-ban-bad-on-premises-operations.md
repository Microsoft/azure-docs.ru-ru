# <a name="enable-on-premises-azure-active-directory-password-protection"></a>Включить локальную Azure Active Directory защиту паролем

Многие пользователи создают пароли с характерными распространенными словами, например названием школы, спортивной команды или именем известного человека. Такие пароли легко угадать и они плохо защищены от атак на основе словарей. Чтобы применить надежные пароли в Организации, Azure Active Directory (Azure AD) Защита паролем предоставляет глобальный и настраиваемый список запрещенных паролей. Запрос на изменение пароля завершается ошибкой, если в списке запрещенных паролей есть совпадение.

Чтобы защитить локальную среду домен Active Directory служб (AD DS), можно установить и настроить защиту паролей Azure AD для работы с локальным контроллером домена. В этой статье показано, как включить защиту паролей Azure AD для локальной среды.

Дополнительные сведения о том, как служба защиты паролей Azure AD работает в локальной среде, см. в статье [как применить защиту паролей Azure AD для Windows Server Active Directory](concept-password-ban-bad-on-premises.md).

## <a name="before-you-begin"></a>Перед началом

В этой статье показано, как включить защиту паролей Azure AD для локальной среды. Прежде чем завершить работу с этой статьей, [установите и зарегистрируйте службу защиты паролей Azure AD и агенты контроллера домена](howto-password-ban-bad-on-premises-deploy.md) в локальной среде AD DS.

## <a name="enable-on-premises-password-protection"></a>Включить локальную защиту паролей

1. Войдите в [портал Azure](https://portal.azure.com) и перейдите к **Azure Active Directory**  >    >  **методы проверки подлинности** защиты  >  **пароля**.
1. Задайте для параметра **включить защиту паролем в Windows Server Active Directory** значение *Да*.

    Если для этого параметра установлено значение *нет*, все развернутые агенты контроллера домена для защиты ПАРОЛЕЙ Azure AD переходят в режим загружена, где все пароли принимаются как есть. Действия проверки не выполняются, и события аудита не создаются.

1. Рекомендуется изначально установить **режим** *аудита*. После того, как вы знакомы с этой функцией и влиянием на пользователей в Организации, можно переключить **режим** на *принудительное*. Дополнительные сведения см. в следующем разделе о [режимах работы](#modes-of-operation).
1. Когда все будет готово, щелкните **Сохранить**.

    [![Включите локальную защиту паролей в разделе методы проверки подлинности в портал Azure](media/howto-password-ban-bad-on-premises-operations/enable-configure-custom-banned-passwords-cropped.png)](media/howto-password-ban-bad-on-premises-operations/enable-configure-custom-banned-passwords.png#lightbox)

## <a name="modes-of-operation"></a>Режимы работы

При включении защиты паролей в локальной среде Azure AD можно использовать режим *аудита* или *принудительный* режим. Рекомендуется всегда запускать начальное развертывание и тестирование в режиме аудита. Записи в журнале событий должны отслеживаться, чтобы определить, будут ли какие-либо существующие рабочие процессы переноситься после включения режима *принудительного применения* .

### <a name="audit-mode"></a>Режим аудита

Режим *аудита* предназначен для запуска программного обеспечения в режиме "что если". Каждая служба агента контроллера домена для защиты паролем Azure AD оценивает входящий пароль в соответствии с действующей в настоящее время политикой.

Если текущая политика настроена в режиме аудита, то "неверные" пароли приводят к сообщениям журнала событий, но обрабатываются и обновляются. Такое поведение является единственным различием между режимами аудита и принудительного применения. Все остальные операции выполняются одинаково.

### <a name="enforced-mode"></a>Принудительный режим

*Принудительный* режим предназначен для окончательной настройки. Как и в режиме аудита, каждая служба агента контроллера домена для защиты паролем Azure AD оценивает входящие пароли в соответствии с действующей в настоящее время политикой. Если включен режим принудительного включения, то пароль, который считается небезопасным в соответствии с политикой, отклоняется.

Когда агент контроллера домена для защиты паролем Azure AD отклоняет пароль в принудительном режиме, он видит похожую ошибку, подобную той, которая могла бы увидеть, был ли пароль отклонен традиционным локальным применением сложности паролей. Например, пользователь может увидеть следующее традиционное сообщение об ошибке на экране входа в Windows или смене пароля:

*"Не удалось обновить пароль. Значение, указанное для нового пароля, не соответствует требованиям домена к длине, сложности или истории ".*

Это сообщение является лишь одним из примеров нескольких возможных результатов. Конкретное сообщение об ошибке может различаться в зависимости от фактического программного обеспечения или сценария, который пытается установить небезопасный пароль.

Затронутым конечным пользователям может потребоваться работать с ИТ-специалистами, чтобы понять новые требования и выбрать безопасные пароли.

> [!NOTE]
> Защита паролем Azure AD не контролирует конкретное сообщение об ошибке, отображаемое клиентским компьютером при отклонении ненадежного пароля.
